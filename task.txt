## Especificação das entidades

Use Lombok igual ao repo dela (`@Getter @Setter @NoArgsConstructor @AllArgsConstructor
@EqualsAndHashCode(of = "id")`), `@Entity`, `@Table(name = "tb_...")`, `@Id` com
`@GeneratedValue(strategy = GenerationType.IDENTITY)`.

**DIFERENÇA OBRIGATÓRIA EM RELAÇÃO AO REPO DELA:** as entidades dela quase não usam `@Column`,
mas a prova EXIGE `@Column` com `name`, `nullable`, `length` etc. em todos os atributos. Então
adicione `@Column` em cada atributo (exceto o id e as relações), seguindo o resto do estilo dela.

### Entidade `Fazenda` — tabela `tb_fazenda` (lado "1")

| Atributo | Tipo Java | Coluna / restrições |
|---|---|---|
| `id` | `Long` | `@Id @GeneratedValue(IDENTITY)` |
| `nome` | `String` | `@Column(name="nome", nullable=false, length=100)` |
| `municipio` | `String` | `@Column(name="municipio", nullable=false, length=80)` |
| `areaHectares` | `Double` | `@Column(name="area_hectares", nullable=false)` — **atributo numérico** |
| `dataCadastro` | `LocalDate` | `@Column(name="data_cadastro", nullable=false)` — **atributo data** |
| `leituras` | `List<LeituraNdvi>` | `@OneToMany(mappedBy="fazenda")`, inicializar `= new ArrayList<>()` |

### Entidade `LeituraNdvi` — tabela `tb_leitura_ndvi` (lado "N")

| Atributo | Tipo Java | Coluna / restrições |
|---|---|---|
| `id` | `Long` | `@Id @GeneratedValue(IDENTITY)` |
| `nomeArea` | `String` | `@Column(name="nome_area", nullable=false, length=100)` |
| `valorNdvi` | `Double` | `@Column(name="valor_ndvi", nullable=false)` — **atributo numérico** |
| `status` | `String` | `@Column(name="status", nullable=false, length=20)` (ex.: SAUDAVEL, ATENCAO, ALERTA) |
| `dataLeitura` | `LocalDate` | `@Column(name="data_leitura", nullable=false)` — **atributo data** |
| `fazenda` | `Fazenda` | `@ManyToOne @JoinColumn(name="fazenda_id", nullable=false)` |

As duas entidades têm 4 atributos + id, com um numérico e uma data cada — atendendo a prova.

## DTOs (replicar o estilo de `CategoriaDto` / `ProdutoDTO`)

Mesmo padrão dela: `@Getter @NoArgsConstructor @AllArgsConstructor`, campos com constraints do
Bean Validation, e um construtor que recebe a entidade e copia os campos (o DTO da entidade "N"
instancia o DTO da entidade "1" dentro dele, igual ao `ProdutoDTO` que cria um `CategoriaDto`).

### `FazendaDto`
- `id` (Long, sem validação)
- `nome` — `@NotBlank` + `@Size(min=3, max=100)`
- `municipio` — `@NotBlank` + `@Size(min=2, max=80)`
- `areaHectares` — `@NotNull` + `@Positive`
- `dataCadastro` — `@NotNull` + `@PastOrPresent`
- construtor `FazendaDto(Fazenda fazenda)` que copia os campos

### `LeituraNdviDto`
- `id` (Long, sem validação)
- `nomeArea` — `@NotBlank` + `@Size(min=3, max=100)`
- `valorNdvi` — `@NotNull` + `@DecimalMin("-1.0")` + `@DecimalMax("1.0")` (NDVI vai de -1 a 1)
- `status` — `@NotBlank` + `@Size(min=3, max=20)`
- `dataLeitura` — `@NotNull` + `@PastOrPresent`
- `fazenda` — `@NotNull` (tipo `FazendaDto`)
- construtor `LeituraNdviDto(LeituraNdvi leitura)` que copia os campos e faz `fazenda = new FazendaDto(leitura.getFazenda())`

## Repositories, Services, Controllers

- **Repositories:** interfaces vazias estendendo `JpaRepository<Entidade, Long>`, igual às dela.
- **Services:** copiar o estilo de `CategoriaService` / `ProdutoService` — `@Service`, `@Autowired`
  em campo, `@Transactional(readOnly = true)` nos finds, `@Transactional` nos save/update, método
  privado `copyDtoToEntidade(...)`, lançando `ResourceNotFoundException` quando não encontra e
  `DatabaseException` em violação de integridade. O `LeituraNdviService` injeta também o
  `FazendaRepository` e usa `getReferenceById` para associar a fazenda, igual o `ProdutoService`
  faz com a categoria. No update, **corrigir o bug do repo dela**: chamar `repository.save(...)`
  antes de retornar o DTO (no `ProdutoService` dela o save foi esquecido no update — no nosso, salve).
- **Controllers:** copiar o estilo de `CategoriaController` / `ProdutoController` —
  `@RestController`, `@RequestMapping("/fazendas")` e `@RequestMapping("/leituras")`, todos os
  métodos retornando `ResponseEntity`, GET-all (`ok`), GET-by-id (`ok`), POST (`created` com URI
  via `ServletUriComponentsBuilder`), PUT (`ok`), DELETE (`noContent`). Use `@Valid @RequestBody`
  nos POST/PUT. No DELETE use `ResponseEntity<Void>`.

## Camada de exceptions (copiar IGUAL, sem mudança de domínio)

Replique exatamente, só trocando o pacote base para `juan_556678.farmeye`:
- `exceptions/DatabaseException` e `exceptions/ResourceNotFoundException` (ambas `extends RuntimeException`)
- `exceptions/dto/CustomErrorDto` (timestamp, status, error, path)
- `exceptions/dto/FieldMessageDto` (fieldName, message)
- `exceptions/dto/ValidationErrorDto` (extends CustomErrorDto, lista de FieldMessageDto, método addError)
- `exceptions/handler/GlobalExceptionHandler` (`@RestControllerAdvice`) tratando:
  - `ResourceNotFoundException` → 404
  - `MethodArgumentNotValidException` → 422 (preenchendo os FieldMessageDto)
  - `HttpMessageNotReadableException` → 400
  - `MethodArgumentTypeMismatchException` → 400
  - `DatabaseException` → 409

   - `GET /fazendas` e `GET /leituras` → 200 com os dados carregados

   - `POST /leituras` válido → 201 com header Location
   - `PUT` → 200; `DELETE` → 204
